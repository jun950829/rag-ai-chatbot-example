# Ingestion Guide (데이터 수집 가이드)

## 목적

이 readme는 RAG 애플리케이션에서 ingestion 레이어를 어떻게 설계해야 하는지
설명.

특정 언어(Python, Node, Go 등)를 정한 것은 아니고\
**동작 방식, 구조, 책임 분리**를 표준화하는 것이 목표.

아래 내용들은 모두 예시 임.

------------------------------------------------------------------------

## Ingestion Layer의 역할

-   원본 데이터 수집
-   데이터 정규화
-   DB 저장
-   부모-자식 관계 유지
-   안전한 갱신(Refresh)
-   검색 및 AI 처리를 위한 준비

------------------------------------------------------------------------

## 일반적인 데이터 구조

-   회사 (companies)
-   제품 (products)
-   FAQ / 일정 (선택)

------------------------------------------------------------------------

## 설계 원칙

### 피해야 할 것

-   하나의 스크립트에 모든 로직을 넣는 것

### 반드시 분리

-   데이터 수집
-   정규화
-   DB 저장
-   삭제 처리
-   인덱싱
-   검증
-   리포팅

------------------------------------------------------------------------

## 추천 파이프라인

### 1. 원본 데이터 스냅샷 저장

-   디버깅
-   롤백
-   변경 비교

------------------------------------------------------------------------

### 2. 데이터 정규화

-   공백 제거
-   빈값 → null
-   Excel 값 처리 (= "2026")
-   JSON 문자열 파싱
-   숫자 변환

------------------------------------------------------------------------

### 3. 부모 → 자식 순서 저장

예: companies → products

------------------------------------------------------------------------

### 4. 중복 방지 (Idempotent Upsert)

-   external_id
-   productId
-   companyId

------------------------------------------------------------------------

### 5. Base Table = 진짜 데이터

저장: - 회사명 - 제품 설명 - 카테고리

저장 금지: - embedding - AI 가공 데이터

------------------------------------------------------------------------

### 6. 인덱싱은 나중에

순서: ingest → prune → index → validate

------------------------------------------------------------------------

### 7. 검증 필수

-   row 수 확인
-   orphan 데이터 확인
-   데이터 손실 여부 확인

------------------------------------------------------------------------

## Refresh 전략

### Full Refresh

truncate → reload → reindex → validate

### Incremental Refresh

upsert → prune → reindex → validate

------------------------------------------------------------------------

## RAG 구조

### Profile

-   엔티티 단위
-   검색 및 랭킹

### Evidence

-   세부 정보
-   정확한 답변

------------------------------------------------------------------------

## Content Hash

-   변경 감지
-   재처리 최소화

------------------------------------------------------------------------

## Batch 처리

-   배치 단위 commit
-   로그 출력
-   일부 실패 허용

------------------------------------------------------------------------

## 금지 사항

-   ingestion + indexing 혼합
-   삭제 로직 숨기기
-   validation 생략

------------------------------------------------------------------------

## 구조 모델

raw → normalized → DB → index → validation

------------------------------------------------------------------------

## 최종 원칙

시스템은 반드시:

-   반복 가능
-   중복 안전
-   관찰 가능
-   재실행 쉬움
-   검증 가능
