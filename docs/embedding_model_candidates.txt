================================================================================
RAG 임베딩 모델 후보 정리 (참고 문헌 / 메모)
================================================================================
작성 목적: CSV 주입 후 RAG용 임베딩 모델 선정 시 비교·검토용
데이터 맥락: ExhibitorsExport_2026.csv 수준(~1만 행), 추후 약 10배 규모 테스트 예정
================================================================================

[선정 기준 요약]
- 언어: 한국어 + 영어
- 입력: 짧은 검색 질의 + 긴 프로필(Profile documents) + 증거 청크(Evidence chunks)
- 규모: 초기 소규모, 이후 중소 규모
- 비용: 가능한 한 저렴하게
- 지연·정확도: 중간 수준 허용
- 배포: HTTP API로 제공, Swagger / OpenAPI 문서, Postman 등으로 검증

[공통 주의사항]
- 쿼리 임베딩과 문서(프로필·청크) 임베딩은 동일 모델·동일 전처리(차원 축소 포함) 권장
- 가격·토큰 상한은 공급자 정책 변경 가능 → 최종 결정 전 공식 문서·가격표 재확인
- 한·영 혼합 도메인은 소규모 골든 쿼리 세트로 검색 품질을 직접 측정하는 것이 안전

================================================================================
후보 1: OpenAI text-embedding-3-small (1순위 권장 시작점)
================================================================================
개요:
- 다국어 텍스트 임베딩. 한국어 전용 모델은 아니나 일반 RAG에서 널리 사용.

장점:
- API 단순, FastAPI 등으로 래핑 시 OpenAPI(Swagger) 노출이 쉬움
- 입력 토큰 상한이 비교적 큼(공식 스펙 기준 약 8,192 토큰) → 프로필·긴 청크에 유리
- 상대적으로 낮은 단가로 알려짐(참고: 약 $0.02 / 1M 입력 토큰 수준 — 공식 가격표 확인)
- 출력 차원 기본 1536; 필요 시 차원 축소로 저장·검색 비용 절감 가능(문서화된 옵션)

단점 / 리스크:
- 한국어 특화 모델 대비 극단적 품질이 필요하면 벤치마크 필요
- 외부 API 의존, 키 관리·레이트리밋

참고 링크(공식):
- https://platform.openai.com/docs/models/text-embedding-3-small
- https://platform.openai.com/docs/guides/embeddings

================================================================================
후보 2: OpenAI text-embedding-3-large
================================================================================
개요:
- text-embedding-3-small 대비 잠재적 검색 품질 상향 여지, 단가 상승

장점:
- 동일 생태계·동일 API 패턴으로 교체 실험 용이

단점:
- 비용이 small 대비 높음(요구사항 “저렴”과 상충 가능)

용도:
- small으로 베이스라인 후, 품질이 부족할 때만 상향 검토

================================================================================
후보 3: Google Vertex AI — Text Embeddings (예: Gecko Multilingual 계열)
================================================================================
개요:
- GCP Vertex의 텍스트 임베딩 API. 다국어 모델 선택지 존재.

장점:
- 이미 GCP·Vertex를 쓰는 경우 IAM·과금·모니터링 일원화
- 모델별 입력 토큰 상한이 다국어 Gecko 등에서 실무적으로 활용 가능한 수준(모델 카드 확인)

단점:
- GCP 종속, 로컬/타 클라우드와 비교 시 설정 공수
- 모델 ID·가격은 시기별로 변동 — 반드시 공식 문서 확인

참고 링크(공식):
- https://cloud.google.com/vertex-ai/generative-ai/docs/model-reference/text-embeddings-api

================================================================================
후보 4: Cohere — embed-multilingual-v3.0
================================================================================
개요:
- 100+ 언어 지향 다국어 임베딩. RAG·시맨틱 검색에 자주 언급됨.

장점:
- 다국어 명시, 배치 호출 등 운영 패턴 문서화

단점:
- 입력 길이 상한이 512 토큰으로 알려져 있음 → “긴 프로필”은 강한 청킹 필수
- 긴 문서를 한 번에 임베딩하려는 요구와 상충할 수 있음

참고 링크(공식):
- https://docs.cohere.com/docs/cohere-embed
- https://cohere.com/pricing (또는 해당 리전/마켓플레이스 가격)

================================================================================
후보 5: Cohere — Embed v4 (텍스트·긴 컨텍스트 필요 시 검토)
================================================================================
개요:
- 후속 세대로 매우 긴 컨텍스트(예: 128K 토큰급)를 내세우는 자료가 있음 — 제품 문서로 최종 확인

장점:
- 긴 프로필을 덜 쪼개고 임베딩할 여지(모델·플랜에 따름)

단점:
- 단가·제공 채널(API 직접 vs AWS Bedrock 등)이 후보 1과 직접 비교 필요
- 멀티모달 등 부가 기능은 본 프로젝트 범위 밖이면 무시 가능

참고:
- Cohere 공식 문서의 Embed v4 스펙·가격표

================================================================================
후보 6: 자체 호스팅 (예: BGE-M3 등) + 자체 임베딩 API
================================================================================
개요:
- 오픈 가중치 다국어 임베딩을 GPU/CPU 서버에 올리고 FastAPI로 동일하게 Swagger 노출

장점:
- 토큰당 외부 과금 최소화(하드웨어·운영 비용으로 대체)
- 데이터 외부 전송 최소화가 필요할 때 유리

단점:
- 배포·스케일링·모델 버전 관리 부담
- 지연·처리량은 하드웨어에 좌우

용도:
- API “형태”는 동일하게 유지 가능하나, 운영 주체가 내부가 됨
